# Requirement Quality Rules

Each requirement shall be traceable, mandatory, atomic, feasible, verifiable, and classified by domain or type.

Flag ambiguous words such as fast, suitable, appropriate, user-friendly, robust, optimized, as needed, sufficient, easy, minimal, normal condition, if possible, and where applicable.

Derived SRS and SWRS items shall be marked as No, Yes, or Partial and include rationale.

## Software Requirement Writing Rules

When writing SWRS items from SRS items:

- Do not copy the system requirement directly.
- Write from the software development perspective.
- Express externally observable software behavior.
- Identify the relevant software boundary.
- Include only software-relevant behavior, constraints, inputs, outputs, interfaces, data, timing, diagnostic, safety, security, configuration, or calibration aspects.
- Preserve traceability to the parent SRS.
- Mark derived or partially derived software requirements and provide rationale.
- Create open issues for missing details instead of inventing them.

Use ISO/IEC/IEEE 29148-inspired principles for software requirements: clear, necessary, singular, feasible, verifiable, traceable, consistent, and appropriate to the software requirement level.
